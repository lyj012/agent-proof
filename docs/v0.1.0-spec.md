# AgentProof v0.1.0 功能规格

## 1. 版本目标

AgentProof v0.1.0 的目标是交付一个 Windows 本地可运行的 Python CLI 最小闭环。

本版本只做一件事：读取用户明确提供的本地证据材料，并生成一份结构固定、客户可读、便于人工审核的 Markdown 交付报告。

v0.1.0 不做智能判断，不自动证明代码正确，不自动判断需求是否完成，也不执行构建或测试命令。

## 2. 产品原则

v0.1.0 遵循以下原则：

- 本地优先：所有输入来自用户明确指定的本地文件或本地 Git 仓库；
- 最小闭环：优先保证 `agentproof generate` 可用，不扩展平台能力；
- 证据分级：区分开发者声明、文件存在、工具实际读取和人工审核；
- 安全默认：默认屏蔽常见敏感信息，不输出不必要的本机绝对路径；
- 客户可读：报告面向客户或项目负责人，不输出过长、过细的技术噪音；
- 人工最终确认：AgentProof 辅助整理证据，但不替代人工验收。

## 3. CLI 范围

v0.1.0 支持以下命令：

```text
agentproof --help
agentproof --version
agentproof generate
agentproof generate --help
```

`generate` 命令参数：

```text
--repo         本地 Git 仓库路径，必填
--task-file    task.json 文件路径，必填
--transcript   AI 开发记录文件路径，必填
--output       Markdown 报告输出路径，默认 delivery-report.md
```

不支持：

- 交互式向导；
- 配置文件；
- 环境变量配置；
- 子命令插件；
- 多报告模板；
- 自动扫描历史目录。

## 4. 输入契约

### 4.1 Git 仓库

用户通过 `--repo` 指定一个本地 Git 仓库。

v0.1.0 只读取最近 1 个 commit。

读取内容：

- commit hash；
- commit message；
- commit author；
- commit date；
- 修改文件列表；
- diff 新增行数；
- diff 删除行数；
- 文件级 diff 摘要。

Git 统计口径：

- 使用 `git show --shortstat --format= HEAD` 获取总文件数、新增行数和删除行数；
- 使用 `git show --stat --format= HEAD` 获取文件级摘要；
- 不默认读取或输出 patch 级完整 diff；
- 如果 Git 输出中缺少新增或删除行数，按 `0` 处理。

不读取：

- commit range；
- 多个 commit 聚合；
- 未提交变更；
- 远程 Pull Request；
- GitHub issue；
- GitHub Actions；
- GitHub API。

### 4.2 任务文件

用户通过 `--task-file` 指定 UTF-8 编码的 JSON 文件。

v0.1.0 推荐结构：

```json
{
  "task_name": "Example delivery task",
  "requirement": "Describe the original requirement.",
  "build_result": "not_run",
  "test_result": "not_run",
  "risks": [],
  "known_issues": [],
  "acceptance_steps": [],
  "manual_review": []
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `task_name` | string | 是 | 本次交付任务名称 |
| `requirement` | string | 是 | 原始需求或任务目标 |
| `build_result` | string | 是 | 构建结果声明 |
| `test_result` | string | 是 | 测试结果声明 |
| `risks` | array[string] | 否 | 已知风险 |
| `known_issues` | array[string] | 否 | 遗留问题 |
| `acceptance_steps` | array[string] | 否 | 客户验收步骤 |
| `manual_review` | array[string] | 否 | 已完成人工审核项 |

`build_result` 和 `test_result` 允许值：

```text
not_run
passed
failed
unknown
```

含义：

- `not_run`：开发者声明未执行；
- `passed`：开发者声明通过；
- `failed`：开发者声明失败；
- `unknown`：开发者无法确认结果。

报告必须明确这些值来自开发者声明，不得写成 AgentProof 实际执行结果。

### 4.3 AI 开发记录

用户通过 `--transcript` 指定一个 UTF-8 编码的纯文本或 Markdown 文件。

v0.1.0 支持：

- 手工导出的 Codex 开发记录；
- 手工导出的 Claude Code 开发记录；
- 用户手工整理的 `.txt` 或 `.md` 开发记录。

v0.1.0 不支持：

- 自动读取 Codex 或 Claude Code 历史目录；
- 解析专有数据库；
- 自动登录任何服务；
- 跨目录搜索所有历史记录；
- 多格式自动识别。

报告中只展示经过脱敏和截断后的 AI 开发记录摘录，不默认输出完整 transcript。

v0.1.0 不调用 AI 模型，因此不生成语义摘要。所谓“摘要”在本版本中指确定性的文本摘录：

- 读取 transcript 原文；
- 先执行敏感信息脱敏；
- 再按固定长度截断；
- 报告中展示截断状态。

### 4.4 输出路径

用户通过 `--output` 指定 Markdown 输出路径。

默认输出：

```text
delivery-report.md
```

输出文件使用 UTF-8 编码。

## 5. 报告结构

v0.1.0 使用固定 Markdown 结构，不提供模板系统。

报告结构：

```text
# 交付报告

## 1. 交付概览
## 2. 原始需求
## 3. Git 证据
## 4. 修改文件
## 5. Diff 摘要
## 6. AI 辅助开发记录摘要
## 7. 构建结果
## 8. 测试结果
## 9. 风险与遗留问题
## 10. 人工审核清单
## 11. 客户验收步骤
## 12. 证据限制说明
## 13. 报告元信息
```

### 5.1 交付概览

包含：

- 任务名称；
- 报告生成时间；
- AgentProof 版本；
- Git 仓库名称或路径摘要；
- 最近 commit hash 短值；
- 构建结果声明；
- 测试结果声明。

### 5.2 原始需求

展示 `task.json` 中的 `requirement`。

### 5.3 Git 证据

展示：

- commit hash；
- commit message；
- author；
- date；
- 证据来源：本地 Git 仓库最近 1 个 commit。

### 5.4 修改文件

展示最近 1 个 commit 涉及的文件列表。

不展示不必要的本机绝对路径。

### 5.5 Diff 摘要

展示：

- 新增行数；
- 删除行数；
- 文件级摘要；
- Git 统计来源说明。

不默认输出完整 patch diff。

实现口径：

- 总增删行来自 `git show --shortstat --format= HEAD`；
- 文件级摘要来自 `git show --stat --format= HEAD`；
- 报告不得包含完整代码 patch；
- 如果某个 commit 没有文本增删行，报告显示 `0 insertions, 0 deletions` 或 Git 实际统计结果。

### 5.6 AI 辅助开发记录摘要

展示：

- transcript 文件名；
- 读取状态；
- 脱敏后的文本摘录；
- 是否发生截断；
- 证据来源：用户指定的本地 transcript 文件。

不展示完整提示词和完整 AI 回复。

实现口径：

- 最大摘录长度为 4000 个字符；
- 超出长度时截断，并显示 `Transcript truncated: yes`；
- 未超出长度时显示 `Transcript truncated: no`；
- 摘录不是 AI 语义总结，不代表 AgentProof 理解或确认了开发记录内容。

### 5.7 构建结果

展示：

```text
Build result: <value>
Source: developer-declared
AgentProof executed build: no
```

### 5.8 测试结果

展示：

```text
Test result: <value>
Source: developer-declared
AgentProof executed tests: no
```

### 5.9 风险与遗留问题

展示：

- `risks`；
- `known_issues`；
- 如果为空，明确显示 `None declared`。

### 5.10 人工审核清单

报告固定包含以下检查项：

- 原始需求与交付内容是否一致；
- 修改文件是否属于本次任务范围；
- 是否存在无关代码修改；
- 是否包含密钥、token、密码、cookie 或其他敏感信息；
- 构建结果是否由开发者确认；
- 测试结果是否由开发者确认；
- 风险和遗留问题是否已经说明；
- 客户验收步骤是否明确、可执行；
- 证据限制是否已经告知客户或项目负责人。

如果 `task.json` 提供 `manual_review`，报告额外展示开发者声明已完成的审核项。

### 5.11 客户验收步骤

优先展示 `task.json` 中的 `acceptance_steps`。

如果未提供，报告显示：

```text
No acceptance steps provided. Developer should add project-specific verification steps before delivery.
```

### 5.12 证据限制说明

必须说明：

- AgentProof 只读取用户指定的本地输入；
- v0.1.0 只读取最近 1 个 commit；
- v0.1.0 不执行构建命令；
- v0.1.0 不执行测试命令；
- 报告中的构建和测试结果来自开发者声明；
- 报告不代表代码一定正确，也不代表需求一定完成；
- 最终验收需要人工确认。

### 5.13 报告元信息

包含：

- AgentProof 版本；
- 报告生成时间；
- 输入文件名；
- 输出文件名；
- 证据级别说明。

## 6. 敏感信息保护

v0.1.0 默认对任务文件、AI 开发记录和 diff 摘要中的常见敏感信息做基础脱敏。

匹配关键词包括：

```text
api_key
apikey
token
secret
password
passwd
authorization
cookie
```

脱敏后的显示：

```text
[REDACTED]
```

v0.1.0 不提供输出原文选项。

v0.1.0 不保证识别所有敏感信息，因此报告仍必须包含人工审核清单，提醒开发者在交付前检查敏感内容。

脱敏实现规则：

- 匹配大小写不敏感；
- 支持 `key=value`、`key: value`、`"key": "value"` 形式；
- 支持 `Authorization: Bearer <value>`；
- 支持 `Cookie: <value>`；
- 只替换敏感值，尽量保留字段名；
- 脱敏后的值统一显示为 `[REDACTED]`；
- Windows 本机绝对路径只保留文件名或仓库内相对路径；
- 无法识别为仓库内相对路径的本机绝对路径，显示为 `[LOCAL_PATH_REDACTED]`。

示例：

```text
api_key=abc123 -> api_key=[REDACTED]
Authorization: Bearer abc123 -> Authorization: Bearer [REDACTED]
C:\Users\name\project\secret.txt -> [LOCAL_PATH_REDACTED]\secret.txt
```

## 7. 证据可信度规则

报告中的信息必须按来源区分。

### 7.1 开发者声明

来自 `task.json` 的内容属于开发者声明，包括：

- 原始需求；
- 构建结果；
- 测试结果；
- 风险；
- 遗留问题；
- 验收步骤；
- 人工审核项。

### 7.2 AgentProof 读取证据

AgentProof 实际读取到的内容包括：

- 本地 Git 最近 1 个 commit；
- 修改文件列表；
- diff 摘要；
- 用户指定的 transcript 文件内容。

### 7.3 AgentProof 未执行事项

v0.1.0 明确不执行：

- 构建命令；
- 测试命令；
- 代码审查；
- 安全扫描；
- 需求完成度判断。

### 7.4 人工最终确认

最终结论必须由开发者或项目负责人确认。AgentProof 只提供整理后的证据和审核清单。

## 8. 生成流程

`agentproof generate` 的处理顺序固定如下：

1. 解析 CLI 参数；
2. 校验 `--repo`、`--task-file`、`--transcript` 和 `--output`；
3. 读取并校验 `task.json`；
4. 验证 `--repo` 是否为 Git 仓库；
5. 读取最近 1 个 commit 的基础信息；
6. 读取修改文件列表；
7. 读取 diff 总增删行和文件级摘要；
8. 读取 transcript 文件；
9. 对任务信息、Git 摘要和 transcript 摘录做基础脱敏；
10. 对 transcript 执行固定长度截断；
11. 生成 Markdown 报告；
12. 写入 `--output`；
13. 在 CLI 输出生成成功信息。

失败处理规则：

- 任一步失败时，命令返回非零退出码；
- 任一步失败时，不输出“生成成功”；
- 输入校验失败时，不生成不完整或误导性报告；
- 如果输出文件已经存在，v0.1.0 允许覆盖；
- 覆盖失败时返回清晰错误。

## 9. 数据校验规则

### 9.1 `task.json` 校验

必填字段：

- `task_name`；
- `requirement`；
- `build_result`；
- `test_result`。

校验规则：

- 必填字段缺失时报错；
- `task_name` 和 `requirement` 必须是非空字符串；
- `build_result` 和 `test_result` 必须是允许枚举值；
- `risks`、`known_issues`、`acceptance_steps`、`manual_review` 缺失时按空数组处理；
- 可选数组字段必须是字符串数组；
- 数组元素为空字符串时忽略；
- 未知字段允许存在，但 v0.1.0 忽略；
- 单个字符串字段建议不超过 10000 个字符，超出时允许截断并在报告中说明。

### 9.2 路径校验

路径校验规则：

- `--repo` 必须存在；
- `--repo` 必须是 Git 工作区；
- Git 仓库必须至少有 1 个 commit；
- `--task-file` 必须存在且是文件；
- `--transcript` 必须存在且是文件；
- `--output` 的父目录必须存在；
- 所有输入文件按 UTF-8 读取；
- Windows 路径允许包含空格和中文。

### 9.3 Git 数据校验

Git 数据校验规则：

- commit hash 为空时报错；
- 修改文件列表为空时允许继续，但报告必须显示 `No changed files detected`；
- diff 统计读取失败时报错；
- 文件路径在报告中使用 Git 相对路径，不使用本机绝对路径。

## 10. 错误处理

v0.1.0 需要对以下情况给出清晰错误：

- `--repo` 不是 Git 仓库；
- Git 仓库没有 commit；
- `--task-file` 不存在；
- `--task-file` 不是合法 JSON；
- `task.json` 缺少必填字段；
- `build_result` 或 `test_result` 值非法；
- `--transcript` 不存在；
- `--transcript` 不是 UTF-8 文本；
- `--output` 的父目录不存在；
- 输出路径不可写。

错误信息应面向 CLI 用户，说明失败原因和需要修正的输入。

## 11. 最小测试矩阵

v0.1.0 至少覆盖以下测试：

| 场景 | 输入 | 期望结果 |
| --- | --- | --- |
| 正常生成 | 有 1 个 commit 的 Git 仓库、合法 task、合法 transcript | 成功生成 Markdown 报告 |
| 非 Git 目录 | `--repo` 指向普通目录 | 返回清晰错误 |
| 空 Git 仓库 | `--repo` 是无 commit 仓库 | 返回清晰错误 |
| task 文件不存在 | `--task-file` 不存在 | 返回清晰错误 |
| task JSON 非法 | `--task-file` 内容不是合法 JSON | 返回清晰错误 |
| task 缺必填字段 | 缺少 `task_name`、`requirement`、`build_result` 或 `test_result` | 返回清晰错误 |
| 枚举非法 | `build_result` 或 `test_result` 不在允许值内 | 返回清晰错误 |
| transcript 不存在 | `--transcript` 不存在 | 返回清晰错误 |
| transcript 超长 | transcript 超过 4000 字符 | 报告截断并显示 `Transcript truncated: yes` |
| 敏感字段脱敏 | 输入包含 `api_key`、`token`、`Authorization` 或 `Cookie` | 报告中显示 `[REDACTED]` |
| Windows 路径 | 输入路径包含空格或中文 | 正常生成报告 |
| 输出目录不存在 | `--output` 父目录不存在 | 返回清晰错误 |

发布前必须至少运行：

```text
.venv\Scripts\python.exe -m pytest
.venv\Scripts\agentproof.exe --help
.venv\Scripts\agentproof.exe --version
.venv\Scripts\agentproof.exe generate --help
.venv\Scripts\agentproof.exe generate --repo . --task-file examples\task.json --transcript <真实记录文件> --output delivery-report.md
```

v0.1.0 不做：

- 登录注册；
- 数据库；
- Web 页面；
- 前后端系统；
- 云端服务；
- SaaS；
- 多租户；
- 团队权限；
- 支付；
- GitHub App；
- GitHub OAuth；
- MCP Server；
- AI 模型调用；
- 自动生成代码；
- 自动修改代码；
- 自动判断代码是否正确；
- 自动判断需求是否完成；
- 自动执行构建命令；
- 自动执行测试命令；
- AI token 和费用统计；
- PDF 或 Word 导出；
- 多套报告模板；
- 数据可视化；
- 读取未提交变更；
- 读取 commit range；
- 自动扫描 AI 工具历史目录。

## 13. 发布验收标准

v0.1.0 发布前必须满足：

- Windows 本地可安装；
- `agentproof --help` 正常；
- `agentproof --version` 正常；
- `agentproof generate --help` 正常；
- `agentproof generate` 可以读取真实 Git 仓库；
- 可以读取 AgentProof 自身仓库最近 1 个 commit；
- 可以读取 `examples/task.json`；
- 可以读取一个真实 transcript 文本；
- 可以生成完整 Markdown 报告；
- 报告包含风险和人工审核清单；
- 报告明确说明构建和测试结果来源；
- 报告明确说明 AgentProof 未执行构建和测试；
- 敏感信息基础脱敏生效；
- 自动化测试通过；
- 使用 AgentProof 项目自身完成一次真实报告生成；
- 发布 GitHub Release `v0.1.0`。

## 14. 后续版本候选方向

以下能力不进入 v0.1.0，可作为后续版本评估：

- 指定 commit hash；
- 指定 commit range；
- 读取未提交变更；
- 读取构建或测试输出文件；
- 更严格的 JSON schema；
- 更完善的 transcript 摘要；
- 多种 AI 工具记录格式；
- PDF 或 Word 导出；
- GitHub PR 信息读取；
- 报告模板自定义。
